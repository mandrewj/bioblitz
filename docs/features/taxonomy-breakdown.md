# Taxonomic breakdown (family / genus)

**Status:** shipped 2026-05-19
**Owner:** —
**Affects:** `lib/store.ts` (schema bump), `lib/sync.ts`, `lib/queries.ts`,
new `app/api/views/[slug]/taxonomy/route.ts`,
new `components/taxonomy-card.tsx`, all `data/*.json` (regenerated).

## Intent

Show how observations distribute across higher taxonomic ranks at a
glance. For Eagle Creek beetles (Coleoptera) the answer is "which
families dominate, and inside the dominant families which genera show
up most?" Helps users spot signal (the curculionid- and chrysomelid-
heavy long tail) and gaps without scrolling the species list.

## UX

New card titled **"Taxonomic breakdown"**, in the main column below the
contributors card.

```
┌─ Taxonomic breakdown ───────────────────────────────┐
│  [ Family ] [ Genus ]            Top 15 of 38      │
│                                                     │
│  Curculionidae    ████████████░░░░░░░  198         │
│  Chrysomelidae    █████████░░░░░░░░░░  142         │
│  Coccinellidae    ███████░░░░░░░░░░░░  104         │
│  Carabidae        █████░░░░░░░░░░░░░░   79         │
│  …                                                  │
└─────────────────────────────────────────────────────┘
```

- **Rank toggle** at the top (segmented control, brand blue when
  active). Default: Family.
- **Horizontal bar chart**, ranks on Y, record count on X. Top 15 of
  the selected rank.
- **Source-mix encoding**: bars are stacked — iNat portion in
  Okabe-Ito orange (`ok-orange`), GBIF portion in brand blue
  (`forest-600`). Matches the rest of the dashboard's source palette.
- **Subtitle** under the toggle: `Top 15 of {N}` where N is the number
  of distinct families / genera observed.
- **Hover tooltip** shows the rank name + total + iNat/GBIF split.
- **No interaction beyond hover** in v1 — clicking a bar to filter the
  rest of the dashboard is a tempting follow-up but out of scope.

## Schema change

Today `StoredOccurrence` only carries the species binomial. We need
family + genus + order on every record.

**Schema bump: `schemaVersion: 1 → 2`.** New required fields:

```ts
interface StoredOccurrence {
  // ... existing fields ...
  taxonOrder: string | null;   // NEW
  taxonFamily: string | null;  // NEW
  taxonGenus: string | null;   // NEW
}
```

Per the shared decision in the AskUserQuestion round: old files
become incompatible until re-synced. They live in git and are
regenerated by `npm run sync -- <slug> --full` once after this lands.

`loadView()` already throws on version mismatch — keep that strict
behavior so a stale checkout fails loudly.

## Sync changes

### iNat (`lib/inat.ts` + `lib/sync.ts`)

iNat returns the full taxonomic ancestry on each observation, but
only if requested. Add the standard `&include[]=...` params (or use
the existing default — verify what the iterator currently asks for).
For each observation walk `obs.taxon.ancestors` (and the obs taxon
itself) and pick the entries whose `rank` is `"order"`, `"family"`,
or `"genus"`. Store the names verbatim.

If the taxon is identified above genus rank, the genus is `null`. If
above family, family is `null`. Acceptable — the bar chart drops null
buckets.

### GBIF (`lib/sync.ts`)

GBIF occurrence records already carry `family`, `genus`, and `order`
directly. Just map them through. No new API calls.

## Implementation sketch

### Query

```ts
// lib/queries.ts
export type TaxonRank = "family" | "genus";

export interface TaxonomyRow {
  name: string;
  records: number;
  inatRecords: number;
  gbifRecords: number;
  species: number; // distinct species in this bucket
}

export function getTaxonomyBreakdown(
  slug: string,
  researchOnly: boolean,
  rank: TaxonRank,
  limit = 15
): { rows: TaxonomyRow[]; total: number; rankCount: number };
```

- Single pass; bucket on `o.taxonFamily` or `o.taxonGenus`.
- Skip null buckets in the result rows; surface their existence in a
  small footnote (e.g., "12 records identified above family rank").
- Sort by `records` desc.

### API route

`app/api/views/[slug]/taxonomy/route.ts` — `?rank=family|genus`,
default `family`. `revalidate: 60`.

### Component

`components/taxonomy-card.tsx` — recharts `<BarChart layout="vertical">`.
Stacked bars (`<Bar stackId="src">` × 2). Brand colors. Tooltip
re-uses the existing Recharts content style.

Toggle wires to `useQuery` with `rank` in the key. Switching is
instant (queries are cached client-side).

## Risks

- **iNat ancestry availability.** If `obs.taxon.ancestors` isn't
  populated for some observations (rare but possible for
  newly-described or unranked taxa), we accept `null` and drop those
  rows from the chart.
- **Genus inferred from binomial** would *seem* like a free
  alternative, but it's wrong for hybrid names, subgenus parens,
  cf./aff. forms, and "Genus sp." records. Stick with the
  authoritative `genus` field from the source.
- **Stacked bar overplotting at small counts.** When iNat or GBIF
  contributes 0 to a bucket, recharts renders a 0-width segment —
  fine. When both are tiny, the bar itself is short; that's an honest
  rendering.

## Open questions

- **Should the "Order" rank be selectable too?** For Coleoptera-only
  views, Order is constant ("Coleoptera"), so the toggle would be
  useless. But for a multi-order view (e.g., "Indiana pollinators"
  later), Order would be valuable. Add later when a multi-order view
  exists.
- **Coloring beyond source-mix.** Could color families by Okabe-Ito
  to give a stable family palette across views. More striking but
  more code. Open question.
- **Drilldown:** clicking a family bar could scope the genus chart to
  that family. Powerful but introduces filter-state plumbing. Out of
  scope for v1.

## Out of scope

- Sunburst / treemap visualization.
- Family → genus → species nested drilldown.
- Filtering the map / species panel by selected rank.
- Conservation status, life history, or other taxon-level
  enrichment.
